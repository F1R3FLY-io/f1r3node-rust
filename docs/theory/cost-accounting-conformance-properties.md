# Cost-Accounting Conformance Property Catalog

**An exhaustive, property-centric enumeration of every assertable obligation that the
cost-accounting specifications place on an implementation — each cited to its spec source,
tagged with how to assert it, and cross-referenced to the artifact that currently asserts it.**

*Companion to, and fine-grained successor of, the coarse traceability matrix in
[`cost-accounting-spec-alignment-audit.md`](./cost-accounting-spec-alignment-audit.md) §2.
For the proof-side narrative see
[`cost-accounted-rho-verification.md`](./cost-accounted-rho-verification.md).*

---

## 0. Purpose & how to use this document

This catalog answers one question: **"Does the implementation satisfy the entire
specification?"** It does so by enumerating *every* obligation as a numbered, checkable
**property** and binding each to the artifact that asserts it.

### 0.2 Property row schema

Each property is one row:

| Column                   | Meaning                                                                                                                      |
|--------------------------|------------------------------------------------------------------------------------------------------------------------------|
| **ID**                   | `CA-P-###` — flat, stable (navigability comes from **Category**, not the number)                                             |
| **Property**             | the assertable statement, phrased "An implementation MUST/SHALL …"                                                           |
| **Spec source**          | document tag + § + **line range** (line numbers are the canonical anchor)                                                    |
| **Modality**             | *how* to assert: Rocq · TLA+(TLC) · Sage · Rust(example/property/loom/fuzz) · Verus · Iris · runtime-GUARD · static-analysis |
| **Covering artifact(s)** | concrete identifier(s) + file, or "—"                                                                                        |
| **Status**               | see legend                                                                                                                   |

### 0.3 Status legend

| Status                | Meaning                                                                                    |
|-----------------------|--------------------------------------------------------------------------------------------|
| ✅ **COVERED**        | ≥1 existing artifact fully asserts it                                                      |
| 🟡 **PARTIAL**        | asserted only in some modality or only on a fragment (stated per row)                      |
| 🔴 **GAP**            | spec asserts it; no artifact does → actionable                                             |
| ⏸️ **DEFERRED**        | asserted by spec/related-doc but a deliberately deferred scope item (cite DR / Greg-model) |
| 🚧 **SCOPE-BOUNDARY** | explicitly out of scope per a decision record / settled negative result                    |
| ➕ **EXCEEDS**        | formalization proves strictly more than the spec requires                                  |

### 0.5 Canonicalization notes (load-bearing — read before using the tables)

1. **Rule-numbering caveat.** The catalog uses the **spec §3.6 (RHO) numbering as canonical**:
   Rule 1 (single sig, whole redex), Rule 2 (compound sig, *split* tokens), Rule 3 (compound
   sig, *combined* token), Rule 4 (split processes, *split* tokens), Rule 5 (split processes,
   *combined* token). **The Rocq development swaps 4↔5** (Rocq `R4` = combined-token,
   `R5` = split-tokens), per `cost-accounted-rho-verification.md` §6. Where a row cites a Rocq
   `ruleN_simulation*`, read the Rocq number through that swap.
2. **§-label / line reconciliation.** Cite RHO/MON **line numbers** as the stable anchor
   (section numbers drifted between drafts: the join section is RHO §3.7 here = historical
   "§4.8" = verification-doc "§4.5"; race-elimination RHO §2.3+§7.5 = "§6.4"; overcharge/refund
   §8.2/§8.6 = "§6.3"; deployment boundary §8.10 = "§6.5"). See audit §1.
3. **Token-source vocabulary.** There is **one** system token; **`phlogiston`** is canonical.
   `REV` is a legacy *name* for the same token (a ledger to unify, not a second currency); avoid
   `F1r3caps` (collides with capabilities). `Pay(τ)` is a *type*, not a second token. See the
   Greg token-source model.
4. **What "assert against an implementation" means here.** A property is satisfied when its
   covering artifact is green under the **local-only** gate scripts
   `scripts/check-cost-accounted-rho-{proofs,tla-invariants,sage,verus,iris,…,ALL}.sh`
   (never CI). For Rust rows, the named test passes under `cargo test`.
