# Stage D — Fee `Exchange` + Collection + `!VB` Economic Loop (approved design)

**Status:** Authoritative, USER-APPROVED design (Plan-agent pass + user confirmation; open decisions resolved below). Spec law: `publications/cost-accounting/cost-accounted-rho.tex` ("Fee conversion" tex:3061-3084, "Persistent validator" tex:3086-3100, §7 funding model, App. B). Conforms to DR-4/5/9/11/13. Builds on landed StageA/B/C + WD-D0/D1/D2. **Binding contract for the StageD implementation agent.**

## D.0 — Centerpiece: cost ≠ fee (the WD-D2 settlement is UNCHANGED)
The D2 settlement debit (`post Σ⟦s⟧ = pre − ΣΔ_admitted`, `CloseBlockDeploy::dual_write_supply`) is the **COST** — per-COMM tokens consumed from the signer's own pool (§7.4), *spent* by the calculus's rewrite rules, never routed to a recipient. **The burn is spec-correct; StageD makes ZERO consensus-critical change to the committed D2 gate/settlement.** The **FEE** is a SEPARATE token — the spec's explicit `FeeExtract` (`F_v!(c:())`, one client token per deploy *transferred* to the validator's fee channel, tex:2509-2521). StageD is purely ADDITIVE: it adds `F_v` collection + `Exchange` + the closeBlock fee→v conversion. **Do NOT touch** `acceptance.rs`'s gate logic, `delta_sigma.rs`, the D2 `settlement_debits` debit, or `TokenConservation.v`'s `settlement_conserves`/`accept_commit_conserves`.

## D.1 — `Exchange` blessed contract (OD-1 join; OD-2 count-datum carriers; OD-5 1:1 peg)
New `casper/src/main/resources/Exchange.rhox`, registered at `rho:lang:exchange`, wired EXACTLY like `capabilities_registry` (`RegistrySigGen::derive_from(&Args::new(Some("Exchange"), Some(EXCHANGE_TIMESTAMP), Some(sk)))` → `$$exchangePubKey$$`/`$$exchangeSig$$` substituted into the `.rhox`; new `EXCHANGE_PK`/`EXCHANGE_TIMESTAMP`/`EXCHANGE_PUB_KEY` in `standard_deploys.rs`, append to `system_public_keys()`, `exchange()` fn, `embedded_rho::EXCHANGE`, `genesis.rs` push after `make_mint` + capacity bump). The contract is the spec's conserving 1:1 swap (tex:3067-3081) as a **persistent join** over ordinary count-datum carrier channels: consumes one `c`-carrier + one `v`-carrier, re-emits one of each swapped (per-channel count preserved). DR-4: the join **requires both inputs** (no one-sided mint). AMM-extensible by construction (variable remainders); **1:1 peg for StageD** (no configurable rate param — avoids a business parameter). Generate `EXCHANGE_PK` as a fixed valid secp256k1 sk hex (hardcoded constant, like `CAPABILITIES_REGISTRY_PK`).

**Carrier vs `Σ⟦v⟧`:** `Σ⟦Ground(pk)⟧` is unnameable from Rholang (no bytes→GPrivate primitive), so Exchange operates on *carrier* channels; the `Σ⟦v⟧` credit is the **Rust `supply::produce_balance` mirror** (same dual-write discipline as the StageB mint).

## D.2 — `F_v` fee collection + closeBlock conversion (OD-3 flat 1-token; OD-4 SETTLED → `Σ⟦v⟧`-only, no `@W_v` mirror — DR-14)
- **`F_v := @(*feeTag, validatorPk)`** — a per-validator PoS-private fee channel; allocate `feeTag` alongside `walletTag`/`quarantineTag` in PoS-state scope. Distinct from `@W_v` (draw) and `Σ⟦v⟧` (gate pool).
- **Fee = flat ONE token per admitted client deploy** (the spec's literal `c:()` `FeeExtract` — NOT `Δ_c` (that's the cost); NOT a configurable param — spec-literal flat, avoids a business parameter), credited to the **proposing** validator (consensus-deterministic: `block_data.sender`).
- **Write seam (replay-symmetric, mirrors D2's `settlement_debits`):** the fee credit to `F_v` is a Rust `produce_balance`-style write on `CloseBlockDeploy::post_eval`/`post_eval_replay`, amount = `count(admitted client deploys in block) × 1`. Play: from `AdmissionOutcome`; replay: recomputed from `block.body.deploys` (identical, like `recompute_settlement_debits`). Use a fresh disjoint `fee_random_state` RNG path. `ReplaySupplyMismatch` readback guard.
- **closeBlock per-epoch conversion (the economic loop, tex:3095-3100):** in the epoch branch, BEFORE the epoch mint, fold over `allBonds`: for each `active ∧ ¬mintingHalted` validator with `F_v` count `f > 0`, run `f` through `Exchange` (1:1) → publish `(v, k=f)` on a `sys:casper:feeConvertList` env channel; `post_eval` credits each eligible `Σ⟦v⟧` via `produce_balance(Σ⟦v⟧, old+k)`; zero `F_v`. **SETTLED (DR-14): `Σ⟦v⟧`-only — NO `@W_v` mirror** (the `@W_v` amount is operationally inert under the s₀-collapse; depositing the fee on `@W_v` would be a consensus no-op). **`convertedEpochs: Set[(Pk,Int)]`** state guard (mirror `mintedEpochs`) ⇒ idempotent under multi-parent merge. DR-4: empty-`F_v` validators get only the epoch MINT, never an Exchange one-sided mint.

## D.3 — `!VB` unchanged
StageA's `VB = for(phlo <= @W_v){*phlo}` (persistent `<=`) IS the spec's `!VB`. No change to VB. The fee-feedback loop = `!VB` re-draw (per deploy) + the closeBlock per-epoch fee-convert (D.2). **Per DR-14 there is NO `@W_v` fee-mirror:** `@W_v` presence (the DR-3 halt anchor) is maintained by the epoch mint, and its amount is operationally inert under the s₀-collapse (the gate enforces `Σ⟦v⟧`).

## D.4 — Client funding / #13 is ORTHOGONAL (do NOT change the gate activation)
StageD provisions the VALIDATOR's `Σ⟦v⟧`/`@W_v` (mint + fee-convert); it does NOT provision CLIENT `Σ⟦c⟧` pools. The D2 gate's transitional `absent ⇒ admit-unenforced` activation (#13) stays exactly as D2 shipped it. Approving/implementing StageD does NOT activate strict client funding (that is #13: genesis flag + client `Σ⟦c⟧` allocation). The blessed `Exchange` is the *acquisition mechanism* clients will use (#13), but StageD only wires the validator loop.
