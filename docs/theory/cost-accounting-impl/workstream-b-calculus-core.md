## Atom-model decision — Option (a) tagged byte-atoms
Replace `SHash : list bool -> sig` (`CostAccountedSyntax.v:76-79`) with:
```coq
Inductive sig := SUnit | SGround : list bool -> sig | SQuote : list bool -> sig | SAnd : sig -> sig -> sig.
```
`SGround`=`g`, `SQuote`=`#P`. **Cost behavior is identical for the two** (every `ca_rule`/conservation/SN/confluence proof treats `s` opaquely; each atom gates one token), so the split is grammar-fidelity (Def 3.3) + the wire `AtomKind`. `FN_s(#P)=FN(P)` is realized at the **source level** via the existing `hash_process : list bool -> proc` bridge — NOT by carrying `proc` in the atom (Option (b) rejected: a `sig→proc` dependency with wide ripple for no theorem the cost-core needs). Mirror `sig_algebra` (`:229`): `ASHash : nat` → `ASGround : nat | ASQuote : nat`, **both → same `LLAtom a`** in `ll_of_sig_algebra` (`LinearLogicResources.v:21`) since `Δ_s` is insensitive to the axis; the `ll_formula` target is unchanged so all DILL proofs re-close verbatim.

## §3.8 sugar — FEASIBILITY VERDICT + decision
The current model (`SSigned : proc -> sig -> system`, `:119`) signs a **bare proc**; `for`/`send` live in `proc` and cannot carry a **signed-term continuation**, so §3.8 (`{for(y<-x){P}_s}_s`) is **not expressible as a `system`-equation**. This is the repo's "extension-of-pure-rho" modeling choice vs the spec's native "signed terms pervade syntax" (§3.1).
- **DECISION (option A, now):** state the sugar at the **translation level** in a new `SyntacticSugar.v` — prove `S_tr(desugar_uniform x P s) ≡ S_tr(sugar_uniform x P s)` (`≡` on `proc` EXISTS, `RhoSyntax.v:719`); likewise lollipop desugars `{for}_ {s₁⊸s₂}` to the `s₁`-outer/`s₂`-inner gate pair using **plain** signatures (never a `Sig::Lolly` node — coexists with the DR-10 extension connective). This discharges the spec's *defining equations* via the spec's own Appendix-A translation.
- **FLAGGED for the user:** the fully-native §3.1 signed-terms model (option B: mutually-recursive proc/system or `SForSigned : name -> system -> sig -> system`) is a calculus-wide reformulation, recorded as a post-B migration. Option A is spec-conformant for §3.8; option B is the spec's surface *presentation* (realized here via translation). No `.tex` change.
