# WD-D2 — Acceptance Gate + Settlement Debit — Authoritative Design

**Status:** Authoritative design (Plan-agent pass, grounded against `feature/cost-accounted-rho @ 694409c8`). **Binding contract for the WD-D2 implementation agent.** Wires three DONE pieces — the pure analyzer `delta_sigma.rs` (WD-D1 `7911fa8b`), the supply helpers `supply.rs` (StageB), and the replay-symmetric `CloseBlockDeploy::post_eval` discipline (`879e75c5`) — into a per-signature acceptance gate + settlement debit. Spec (LAW): `publications/cost-accounting/cost-accounted-rho.tex` §7.6/§7.7 (accept-then-execute, reject-both), Appendix A. Supersedes the workstream-d-acceptance.md D2 sketch where they differ.

## D2.1 — Gate placement
Add `admit_by_funding` in a NEW `casper/src/rust/util/rholang/acceptance.rs`; call it from `block_creator.rs::create` **after `compute_parents_post_state` (block_creator.rs:790)** and **before execution (`compute_deploys_checkpoint_cosigned`, block_creator.rs:1012)**. This satisfies the spec's gate-before-execute (tex 1726-1729) exactly. **Deviation from the doc's literal "inside `prepare_user_deploys`":** that function has no merged pre-state and no runtime handle (block_creator.rs:75-84) — read "prepare_user_deploys" as *the deploy-preparation phase of `create()`*. The canonical ORDER is still produced by `prepare_user_deploys` (block_creator.rs:314-324: `valid_after_block_number`, then `time_stamp`, then `sig`); but `create()` collects deploys into a `HashSet` (block_creator.rs:946) whose iteration order is **nondeterministic across nodes**, so the gate MUST re-impose the canonical sort itself before grouping/prefix selection. Gate ONLY `user_deploys` (exempt proposer `dummy_deploys`); union gate-rejected sigs into `rejected_deploys` at packaging.

## D2.2 — `admit_by_funding` algorithm
`pub async fn admit_by_funding(deploys: Vec<Signed<DeployData>>, pre_state_hash: &StateHash, runtime_manager: &RuntimeManager, margin: i64) -> Result<AdmissionOutcome, CasperError>` where `AdmissionOutcome { admitted: Vec<Signed<DeployData>>, rejected: Vec<Bytes>, debits: BTreeMap<SigKey, SettlementDebit> }`, `SettlementDebit { channel: Par, amount: i64 }`.
1. **Canonicalize** the input by the block_creator.rs:315-324 comparator (verbatim).
2. Per deploy (canonical order) build a `Candidate { deploy, sig_key, demand }`:
   - **Envelope `Sig`** via an EXTRACTED shared helper `Cosigned::envelope_sig` (refactor of mod.rs:890-961 + runtime.rs:1487-1498 into ONE function so gate, runtime-install, and replay never drift): single-signer ⇒ `Sig::Quote(Blake2b256(DEPLOY_SIGNATURE_DOMAIN ‖ sig))`; compound ⇒ left-assoc `Sig::And` fold of `Sig::Quote(Blake2b256(COMPOUND_DEPLOY_SIGNATURE_DOMAIN ‖ sig_i))`. (Under s₀ collapse the envelope only drives `sig_key` = the group key + supply channel; getting it wrong mis-keys the pool, so it MUST match the runtime install.)
     > **⚠ SUPERSEDED by §D2.9 (the funding-key correction).** Keying the supply pool by `envelope_sig` (the per-deploy **wire signature** hash) was the bug: genesis seeds wallets at `Σ⟦Sig::Ground(pk)⟧` (the **public key**), so `Σ⟦envelope_sig⟧ ≠ Σ⟦wallet⟧` and no deploy ever debited its wallet. The supply pool is now keyed by `funding_sig` (the signer's GROUND public key), so `Σ⟦signer⟧ == Σ⟦wallet⟧`. The `envelope_sig` derivation survives ONLY as the (unchanged) `deploy_id` basis. See §D2.9.
   - `sig_key = delta_sigma::sig_key(&envelope) = envelope.lane_hash()`.
   - `Δ_s = delta_sigma::demand(&desugar_for_funding(&Compiler::source_to_adt(&deploy.data.term)?), &envelope).known_lower_bound` — use **`known_lower_bound`, NOT `comm_node_count`** (D1→D3 handoff: matches the current per-SourceStep runtime; `comm_node_count` would over-admit). Carry the `unknown` flag into `is_funded`.
   - Malformed term (`source_to_adt` error) ⇒ REJECTED (the runtime would fail it too).
3. Group into `BTreeMap<SigKey, Vec<Candidate>>` (deterministic group iteration; per-group `Vec` preserves canonical order).
4. Build `Decomposition { compound, left, right }` (lane hashes) for each compound `Sig::And` envelope (per internal `And` node for n≥3). `Threshold/Plus/With` ⇒ no decomposition (the runtime forms only `And` today).
5. **Per-group supply read ONCE**: `Σ_s = supply::decode_balance_datum(&runtime_manager.get_data(pre_state_hash.clone(), &supply_channel(&envelope)).await?)`. (Use `get_data` — gate has a hash, not a live `RuntimeOps`; same decoder as `read_balance`.) Read each distinct channel (groups + decomposition components) exactly once into `raw: BTreeMap<SigKey,i64>`.
6. `effective = delta_sigma::effective_supply_with(&raw, &decompositions)`.
7. **Per-group prefix admission (reject-both)**: `residual = effective.get(&sig_key)`; for each candidate in canonical order, admit iff `is_funded(&demand, residual, margin)` (folds margin + `unknown` over-approx), then `residual -= known_lower_bound`. On the FIRST candidate that does not fit, reject it **and all after it in the group** (§7.7 no-partial, tex 1696-1712). Accumulate `ΣΔ_s` over admitted → `debits[sig_key] = SettlementDebit{ channel, amount: ΣΔ_s }`.
8. Return; feed `admitted` to execution in canonical order. **No global lock/barrier** — groups are independent.

## D2.3 — Settlement debit (load-bearing — without it Σ_s never decreases)
`post Σ⟦s⟧ = pre − ΣΔ_admitted` (handoff Decision 4c). Implement as a replay-symmetric Rust `supply::produce_balance` debit **inside `CloseBlockDeploy::dual_write_supply` (close_block_deploy.rs:76), AFTER the mint loop** (closeBlock is always the last system deploy, so all user deploys have executed). Reuse `post_eval`/`post_eval_replay` (close_block_deploy.rs:207,232) — symmetric by construction.
- **Amount** = the gate's static `ΣΔ_s` (NOT runtime counters) — `replay_cost_mismatch` already proves `runtime consumed == Δ_s` per deploy, so the static sum equals the actual consumed total and is bit-identical play/replay.
- **Transport:** play threads `AdmissionOutcome.debits` → a new `CloseBlockDeploy.settlement_debits` field (populated at block_creator.rs:917; defaults empty for genesis/non-cost-accounted). **Replay RECOMPUTES** `ΣΔ_s` from `block.body.deploys` (the admitted full `Signed<DeployData>`s) — the same deterministic function — and feeds the same map to `post_eval_replay`.
- **Write:** `old = read_balance(chan); new = old.checked_sub(amount).expect("supply underflow — gate invariant violated"); produce_balance(chan, new, debit_random_state(close_rand, idx))`. Use a **disjoint RNG path** from the mint loop (`supply::debit_random_state`, mirroring the mint's `split_byte(0x2a)`) so mint+debit to a shared channel get distinct datum identities; read-modify-replace keeps the trie root deterministic. Replay adds the `ReplaySupplyMismatch` write-readback guard.
- **Compound-debit deferral (TRACKED FOLLOW-ON → D3/funding-slots):** for compound envelopes the Split/Join credit (`effectiveΣ = Σ_compound + min(Σ_l,Σ_r)`) could exceed the compound's own pool. D2 **debits each lane's OWN channel only** and correspondingly **caps compound admission at `Σ_compound`** (component-pair credit treated as non-spendable in D2) — safe (never admits unfunded / never underflows), conservatively under-live for multi-sig (rare; the runtime forms only `And`). The full multi-pool draw-allocation is a funding-slot mechanism, out of D2's consensus-gate scope. **Single-signer (the only shape the pool carries today, all §7.4 examples) is EXACT: `Σ⟦s⟧ -= ΣΔ_s`.**

## D2.4 — Replay determinism: `ReplayAdmissionMismatch`
Add `ReplayFailure::ReplayAdmissionMismatch { expected_admitted, replay_admitted, expected_rejected, replay_rejected, detail }` + constructor + `Display` (sibling of `ReplayCostMismatch`/`ReplaySupplyMismatch`, replay_failure.rs). In `replay_deploys` (replay_runtime.rs:111), after reset and BEFORE the deploy loop, recompute the gate over `block.body.deploys` (admitted, full deploys present) against `start_hash` + genesis margin and assert: **(a)** every admitted deploy is fundable in canonical order against the recomputed group residuals (catches a proposer admitting an unfunded deploy = double-spend, TM-CA-153); **(b)** the reject direction relies on the **post-state root** (rejected-deploy bodies are not in the block, only sigs — intentionally not re-normalized; a wrongly-rejected fundable deploy changes the admitted set → diverges the root, caught by the existing root check); **(c)** `ΣΔ_s` recomputed == the debit applied (same map → identical). Composes with `replay_cost_mismatch` (per-deploy consumed==Δ_s) + `replay_supply_mismatch` (debit readback) — the three guard the three views (pre-state read, in-pass residual, post-state balance).

## D2.5 — Genesis margin
Reuse on-chain `CasperShardConf.min_phlo_price` (casper.rs:347; in `casper_snapshot.on_chain_state.shard_conf`) as the gate `margin` — the natural successor to the legacy minimum-phlo-price floor, on-chain and replay-deterministic. **No new genesis param for D2** (add a dedicated `funding_margin` only if D3 decouples it). Δ_s and Σ_s share one integer unit — no separate resolution constant.

## D2.6 — Speculative-execution-on-receipt: SCOPE DECISION → D2-perf follow-on
**D2 does NOT implement it; the gate alone is fully spec-conformant.** The spec mandates accept-then-execute (tex 1726-1729), which the gate provides (no admitted deploy executes before the funding decision; rejected deploys never execute). The spec does NOT mandate speculative execution-on-receipt — it is a latency optimization, and DR-11's "gate-before-speculate" is a *constraint on any speculation* (must not feed acceptance/commit), not a *requirement to speculate*. Today there is NO execution-on-receipt (deploys sit in storage until a proposer picks them). Adding it is a **large architectural change** (new execution trigger at ingress + soft-checkpoint + a `committed` I/O gate + speculative-cache lifecycle), touching ingress/runtime/I/O — orthogonal to the gate's correctness (a pure O(AST) static analysis needing no speculative results). **Tracked as D2-perf** (the `create_soft_checkpoint`/`revert_to_soft_checkpoint` machinery already exists for it). Nothing consensus-critical is deferred.

## D2.9 — Funding key = the signer's GROUND public key (`Σ⟦signer⟧ == Σ⟦wallet⟧`)

**Status:** IMPLEMENTED + verified (this correction supersedes the §D2.2 `envelope_sig` keying). **Consensus-critical** — changes which pool every deploy debits. Spec basis: `cost-accounted-rho.tex` §"signature grammar" (`g` = *"an Ed25519 public key, a secp256k1 key hash"*) + §1485-1495 (per-actor pools) + §1613 (a deploy consumes its signer's pool); corroborated by `typed_value.tex`, `reputation.tex`, `continued-gslt-cost-v2.tex`, `rent_and_shard_splitting.tex`, `proofs-as-processes-continued.tex`; refuted by none. Confirmed directly by Greg + Mike.

### The bug
A deploy's funding pool was keyed by `envelope_sig` = `Sig::Quote(Blake2b256(DOMAIN ‖ wire_sig))` — the per-deploy **wire signature**, a fresh value every deploy. Genesis seeds wallets at `Σ⟦Sig::Ground(pk)⟧` (the signer's **public key**; `close_block_deploy.rs:228/514`). The two channels never coincide (`hash(wire_sig) ≠ pk`), so a deploy's pool was always ABSENT ⇒ the gate's absent-pool branch admitted it unenforced with no debit ⇒ **the wallet was never debited** and the linear `Σ ≥ Δ` proof never bound.

### The fix — decouple `deploy_id` from the funding key
`envelope_sig` is overloaded: it derived BOTH the `deploy_id` (needs per-deploy uniqueness — the wire sig gives that) AND the supply key (needs per-signer identity — the pubkey gives that). Split them:
- **`deploy_id`** stays **wire-sig-derived** (`envelope_sig_single`/`_compound`) — byte-identical to before; on-chain deploy identity never moves.
- **The funding key** becomes **`funding_sig`** = `Sig::Ground(pk)` for single-sig, `Sig::And` fold of `Sig::Ground(pkᵢ)` for multi-sig — keyed by the signers' GROUND public keys, so the pool the gate reads / proves `Σ ≥ Δ` against / debits IS the genesis-seeded wallet `Σ⟦Ground(pk)⟧`.

The runtime install gains `set_deploy_signature_funded(wire_sig, funding_sig)` / `set_deploy_signatures_funded(...)` (`accounting/mod.rs`): `deploy_id` from the wire sig, `self.signature`/signer-channels from `funding_sig`. The legacy `set_deploy_signature(s)` are now thin wrappers passing the wire-sig `envelope_sig*` (so every test/bench caller is byte-identical). `evaluate_cosigned` (`runtime.rs`) derives `funding_sig(cosigned)` and calls the `_funded` variants. The gate (`build_candidate_with_logic`) and the replay recompute (`recompute_settlement_debits_with_logic`) both key by the SINGLE shared `accounting::funding_sig` — the no-drift guarantee; replay reconstructs the full verified cosigner set via `Cosigned::to_cosigned()` ⇒ byte-identical settlement-debit map.

### Security — the placeholder filter (R1-F4)
`funding_sig` excludes empty-`sig` PLACEHOLDER cosigners (the un-signed members of an M-of-N threshold envelope, `from_signed_data_threshold`). Without this, a deploy could key funding to an UNSIGNED victim's pubkey wallet. The FILTERED funder count (not `is_compound()`) drives the funding arity, so a 1-of-2 threshold with one real signer + one placeholder funds ONLY the real signer's wallet. (Ingress `from_proto_cosigned` already verifies every non-placeholder `sig` against its `pk`.)

### Multi-sig (P8-balanced) + the strict-compound recompute fix
A multi-sig deploy funds from the cosigners' wallets `Σ⟦Ground(pkᵢ)⟧`, balanced (each cosigner debited equally — a compound token is a matched pair, one from each pool; ratified P8). Genesis seeds the individual pubkey wallets, NOT the compound `Σ⟦And(…)⟧` pool, so under enforcement the compound group funds from `effectiveΣ_compound = Σ⟦compound⟧(absent ⇒ 0) + min(Σ⟦left⟧, Σ⟦right⟧)` (the Split/Join component fold).

This exposed a **latent pre-existing bug** (never hit before, because pre-fix all pools were absent so strict rejected everything consistently): the replay recompute's STRICT re-verification + settle-filter keyed on the compound pool's **raw presence** instead of its **effective supply**. Now that component wallets are genuinely funded, a strict compound deploy was play-admitted (effective fold) but recompute-rejected (raw-absent compound) — a play/replay FORK on every strict multi-sig deploy. Fixed: the strict re-verification rejects iff `effective_supply_with(raw, decompositions)[key] ≤ 0` (not raw presence), and the settle-filter keeps a group iff `strict || present(own pool)` — mirroring the play side's early-admit condition exactly (non-strict behavior byte-identical).

### Migration (so nothing breaks)
- **Non-strict shards (default) + empty `client_fuel_allocations`:** every user deploy keys to `Σ⟦Ground(pk)⟧`, which is absent ⇒ early-admit unenforced, no debit ⇒ **post-state byte-identical to pre-fix.** No genesis/integration golden moves.
- **Strict shards:** seed `Σ⟦Ground(client_pk)⟧` via `client_fuel_allocations` (already pubkey-keyed) — this is where the linear `Σ ≥ Δ` proof + data-dependent precharge actually bind.
- **Genesis (block 0) + all system deploys** route through `evaluate_system_source` (not `evaluate_cosigned`), bypass the gate, never install a funding sig — the s₀ always-funded path. Unchanged.
