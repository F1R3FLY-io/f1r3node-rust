## Findings
- **Original "11 warnings" count was wrong**: it came from `cargo clippy --workspace --all-targets`. The pre-commit hook does NOT use --all-targets, so test code isn't linted. Actual hook-gated warnings: 2 (both `needless_return`, both in if-blocks guarding `#[cfg(target_os="linux"...)]` malloc_trim machinery).
- **`cargo clippy --fix` produced semantically broken output**: removed the `return;` but left an empty `if !n.is_multiple_of(interval) { }` block, which on linux would make malloc_trim run UNCONDITIONALLY (losing the every-Nth-block throttle). Reverted via `git checkout HEAD --` and applied manual fix.
- **Root cause is a regression** from b851666a ("new-repo native infrastructure import" on May 12): the prior fixes (`30b0c1c3` for node, `56384d4f` for casper, both Apr 30) had cfg-gated the entire malloc-trim machinery + provided non-linux no-op stubs. The new-repo import wholesale-replaced these files and dropped the gates — same regression class as the b851666a-dropped `ulimit -n 65536` lines that EPOCH-002 captured (now obsolete because dev's a5401484 brought ulimit back via the merge).
- **Correct fix pattern** (applied):
  - Gate `MALLOC_TRIM_INTERVAL_BLOCKS` const, `MALLOC_TRIM_BLOCK_COUNTER` static under cfg
  - Gate the entire `maybe_trim_allocator_after_block` function under cfg, with a no-op stub for non-linux
  - Keep imports unconditional only when used elsewhere (casper's `AtomicU64`/`Ordering` are used by `casper_buffer_last_prune_ms`; node's `AtomicUsize`/`Ordering` were ONLY used by `PROCESSED_BLOCKS`, so the import was also gated)
- Test result: 0 regressions. casper: 383 passed, 31 ignored. node: 107 passed. Other test files: all pass.
